# CONFIG.md - The Configuration System
Principle: if something can change, it changes in exactly one YAML file in config/: never in code.

## Layout (one file per subsystem)
app.yaml (product, branding, feature flags, locales) · ai.yaml (chains, models, cache, quotas, embedding_dim) · auth.yaml (JWT TTLs, activation, anomaly policy, session) · database.yaml (pool, BKT params, decay, diagnostic draw composition) · storage.yaml (provider, dirs, retention) · logging.yaml · cache.yaml (memory|redis) · analytics.yaml (thresholds) · email.yaml (none|resend|gmail_smtp - gmail_smtp live as of M0, SMTP_HOST/PORT/USER/PASSWORD in .env; no sending code until M1) · security.yaml (assessment integrity flags) · ocr.yaml + offline.yaml (Phase 2, pre-built) · deployment.yaml (domains, CORS).

**app.yaml's `branding` block (design-token-system foundation work, between M4 and M5 - docs/DESIGN.md is the full write-up):** `active_palette` and `active_typography` are the two switches - each names a key into `branding.palettes` (8 full color palettes) / `branding.typography` (2 font pairings) respectively, and changing either is the entire rebrand: no code reads a palette or pairing by name, `apps/web/src/lib/config.ts` always resolves whichever is currently active. The block also carries `branding.status` (shared success/warning/error tokens, each with a separately-contrast-checked `*_text` variant - docs/DESIGN.md explains why both exist), `branding.neutral_tracks` (warm/cool 6-step scales), `branding.spacing`/`radius`/`shadow` (the non-palette-scoped scales), and `branding.logo.path`. All of it ships in one `/config.json` response - `core/config.py`'s pydantic `Branding` model validates the shape (schema only), `public()` serves it verbatim.

**ai.yaml's provider comparison (4 providers onboarded - groq/gemini/cerebras/deepseek; claude stays onboarded but deliberately excluded from this comparison, same as production leaving ANTHROPIC_API_KEY unconfigured since M4):** `ai.ai_primary_provider` and `ai.ai_fallback_provider` are the two switches - each names one of the 4 providers, and changing either is the entire decision: `apps/api/app/ai/gateway.py`'s `_resolve_dynamic_chain()` builds `item_bank`/`session`/`dialogue`/`fast_text`'s real provider order as `[primary, fallback, <remaining two in fixed order>]` at call time, so the other two always serve as deeper fallback even when not primary/secondary - no code change needed either way. This is where the final provider decision gets made once real output quality has been compared across all 4 (currently `groq`/`gemini` for prototyping, per the user's own plan, ahead of a final choice closer to the hackathon). `embeddings`/`ocr` chains stay static in `ai.chains` (embeddings needs gemini's dedicated `embed()` capability no other onboarded provider implements; ocr is dormant/vision-specific) - the primary/fallback mechanism only governs the 4 general text-generation chains. Each provider's `ai.models.<name>` entry carries its own `temperature` (kept low, 0.2-0.3, for deterministic structured JSON - this is factual content generation, not chat) with the reasoning documented per-provider in `apps/api/app/ai/providers/*.py`'s module docstring.

## Loading
Backend: app/core/config.py merges config/*.yaml -> Settings; validates at boot (invalid = refuse to start). Frontend: fetches /config.json (secret-free projection: product, branding, features, locales); branding colors injected as CSS variables, so rebranding = editing app.yaml.
Env overrides: `APP__<top>__<nested>=value` (e.g. `APP__ai__cache__demo_mode=true`). Secrets NEVER in YAML: .env only (see .env.example).

## Rules
Changing AI provider order = ai.yaml. Changing brand = app.yaml. Changing retention = storage.yaml. Enabling OCR/offline later = flipping features flags + their pre-built yaml. If a change requires editing code AND config, the design is wrong: fix the code to read config.

## Environments
Local: repo config/ + local .env. Production: same config/ (shipped in the image) + droplet .env + any APP__ overrides in the compose env. Divergence between environments is expressed ONLY through .env and APP__ overrides, never by editing YAML per environment.
