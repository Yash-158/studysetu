**Environment gotcha found this session (WSL2 + Docker, not a code bug - record before it wastes another session):** on this machine, `docker compose -f infra/docker-compose.dev.yml down -v && up -d` does NOT reliably give a truly fresh Postgres: a prior container's `postgres` process was found still running (verified via `ps` + its cgroup path pointing at an old, no-longer-listed container ID) and still answering on host `127.0.0.1:5432` with the OLD data/schema, even though `docker ps -a` no longer listed that container and a new one reported healthy on the same port. `docker compose exec` (which always targets the currently-tracked container) and raw TCP-to-localhost (which apparently could reach the orphaned process) can therefore show DIFFERENT schema/data for a window after a down/up cycle. No passwordless way was available this session to kill the orphan (permission denied even under `sudo -n`). **CD lesson (permanent, not a one-off fix - re-check this every time a new service or script dependency is added):** when a new service is added to `infra/docker-compose.yml`, CD's file-sync step (the `scp-action` steps in `deploy.yml`) must be checked for completeness, never assumed to auto-propagate. During M1's merge, `deploy.yml` shipped `site/*` and `nginx.conf` but not `docker-compose.yml`/`deploy.sh` themselves, so the droplet silently ran M0-era compose/deploy.sh with no `web` service at all for a full extra day after the M1 code merged - CI was green the whole time, only a live CD-run inspection caught it. General rule: **any file a deploy script depends on that isn't baked into the pushed image must have an explicit scp/sync step**, it does not "just come along" with the parts that are already wired. Two smaller CD gotchas from the same debugging session, also worth remembering: (1) a fixed `sleep N` before a container healthcheck is a race, not a wait - poll in a retry loop instead, since a legitimately healthy but slower-starting image (more imports = slower cold start) will false-fail a fixed sleep; (2) `wget`/curl healthchecks against `localhost` inside a container can silently prefer the `::1` IPv6 entry in `/etc/hosts` over `127.0.0.1` - if the service only binds IPv4 (e.g. an nginx image whose entrypoint can't patch a read-only-mounted config to add an IPv6 listener), target `127.0.0.1` explicitly rather than `localhost`. **Known limitations / architectural decisions from M1 (still true, carried forward):**
- **Refresh tokens are stateless JWTs with no revocation path.** No `sessions`/`refresh_tokens` table exists (M1's Modify list didn't authorize a new migration), so there is no way to invalidate a single refresh token, force a logout, or list a user's active sessions - a leaked refresh token stays valid until it naturally expires (`auth.jwt.refresh_ttl_days`, currently 30). Real session tracking (a `sessions` table keyed by jti) is the fix; add it when logout-everywhere, device management, or actual revocation becomes a real requirement - likely M2+ territory since it needs a migration + DATABASE.md update.
- **Concurrent-login anomaly detection is a heuristic, not real session tracking**: a re-login inside the *previous* login's still-valid access-token window (`last_login_at` + `auth.jwt.access_ttl_minutes`) is treated as "concurrent" and emits a `login_anomaly` warn event. This is deliberately approximate (same limitation as above: no sessions table to know for certain two devices are simultaneously active) but matches OQ7's warn-only, not-airtight intent.
- **Activation code TTL is measured from `users.created_at`**, not a dedicated `issued_at` column (none exists). Correct today because M1 only ever sets the activation code at account-creation time; breaks once M2's reissue flow lets an admin issue a fresh code to an already-existing account - M2 should add `issued_at` then.
